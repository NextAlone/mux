---
title: Multi-model fusion
description: Run several selected models in parallel and synthesize one answer
---

Fusion sends the same task to two or more models in parallel, then asks a judge model to combine the results. The final report keeps agreement, disagreements, model-specific insights, blind spots, and a concrete recommendation separate.

Use it for architecture decisions, unfamiliar debugging, reviews, research, or other work where a second independent reasoning path can change the answer. Routine edits usually do not justify the extra latency and token cost.

Configure the default panel (two to eight models) and judge in **Settings → Fusion** first. Fusion refuses to run without these defaults.

Then invoke the built-in skill with only the task:

```text
/fusion Compare the smallest safe approaches to this refactor.
```

You can temporarily replace the models in ordinary language:

```text
/fusion Compare the smallest safe approaches to this refactor. Use Mimo and Gemini to review; use GPT as judge.
```

“Use A and B” replaces the panel for that run, while “also add C” appends to the saved panel. Naming a judge replaces only the judge for that run. These overrides never update Settings; the next run starts from the saved defaults again. You can use model aliases, unique configured provider/model names, or full `provider:model` IDs.

Panel agents have read-only workspace access, so they can inspect evidence without making concurrent edits.

Fusion runs as a durable workflow. Its panel steps execute concurrently, completed steps survive interruption, and the workflow card keeps each child report available for inspection.

## Model-built workflows

Mux also lets the active model construct a one-off workflow directly. Ordered processes are generated as [declarative Markdown templates](/agents/workflows); JavaScript is reserved for parallel lanes, loops, branching, or nested flows.

```text
Build and run a three-stage workflow that analyzes this change, implements it, and verifies the result.
```

The compiled workflow is snapshotted with the run, so restart/resume does not depend on regenerating the template. Reusable team workflows should be saved as a skill-packaged `workflow.md` or an explicit workspace `.md` file.
